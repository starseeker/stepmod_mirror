While in GE

 Eurostep logo 

To eliminate the logo and  incorrect title, cover with a rectangle in VISIO.

or, alternatively  after exporting as XML, crop the *.gif images (see below).

Required comments

The best way to get to get static comments into arm.exp or mim.exp , is to go to the schema page and click on main schema, then add them to the description field.

Example: 

  $Id: GE_guidelines.htm,v 1.4 2002/07/24 00:22:06 thendrix Exp $
  N773 - ISO TS - 10303-1042 Work request - EXPRESS MIM

NOTE : in above, any text after the colon and before the second dollar sign was added by CVS - I did not put it in... This is how it is supposed to work.

Remark  delimiters will be added to Express by GE export process.

Cleanup of files after running ge2module.wsf.

.exp files: 

Remove language version identifier if any.

Remove Eurostep comments.

Remove any extra schemas.

Add N number and cvs keyword - note : these can be added to the GE model once instead of in outputs each time you must modfiy.

 Express-G diagrams 

armexp?.gif and mimexp?.gif

If Eurostep logo is still visible, 

  • crop the gif images, ( for example with  Microsoft Photo Editor). Be sure to crop only bottom and right side, retaining original settings for top and left side. This will assure that image maps still work. 

arm.xml and mim.xml

Add  to express element the attribute description.file = "arm_descriptions.xml"   or "mim_descriptions.xml" 

armexpg?.xml or mimexpg?.xml

If there is no entity level diagram for the mim schema (or occasionaly the arm schema) , replace the href in the image map with an href to the mim listing in clause 5. 

EXAMPLE  For Shape_property_assignment_mim: href="../shape_property_assignment/sys/5_mim.xml#mim_express"

Using GE 1.3.9 - tips to work around stability problems

(From Rob)

Hi
I have had several emails about GraphicalExpress being unstable.

The software is not 100% reliable.

Having used it for a while I now know where some of the bear traps are.

I don't think that the express import works that well - particularly on files with multiple schema.

Avoid the temptation to copy between different Visio diagrams - this can lead to the model being corrupted. Particularly if you copy USE FROM or PAGE REFERENCEs 

Visio will nearly always crash if you publish as XML, then save as express. It successfully outputs the express then crashes. 

Editing the USE relationships between schema on the schema page can cause problems if you have selected the "Explicitly interface items" 

If you resize the pages to make the model fit the smallest page, the image maps will be correct on Windows2000 - I am not sure what happens in other operating systems. NT for example will produce an offset image map. If you are using NT I would recommend leaving the page size as default and position the diagram at the top of the page so that it is visible in the module.

I always save my files a new version frequently so that I can go back to a previous version if the model gets corrupt. i.e. part_arm_v1.vsd 
part_arm_v2.vsd 
part_arm_v3.vsd ....... etc 

I never use the description facility in GraphicalExpress to describe the model. I always run publish my model as XML then run stepmod/utils/extractDescriptions.wsf to get a template for the descriptions in a module i.e. arm_descriptions.xml mim_descriptions.xml, then write them there.  The reason for doing this is that if I corrupt the model, it means that I still have the descriptions. Also, people other than me can update the descriptions.

Finally, there scripts for extracting the XML for modules and the help files that explain how to do this are at: http://homepages.nildram.co.uk/~robbod/

One other area where GE is sometimes unstable is the renaming of schemas. I have found that this sometime crashes GE.  My approach is to try to get me schema names correct first time ;-) If I do have to rename the schemas I rename them one at a time, saving into a new file each time so that I can roll back if something goes wrong.

(From SYL) 

Along this line, if one needs to correct the names of USE_FROM schemas, one should first undo all the Entities referencing the schemas and then do the corrections. Think of Visio as a historical machine like old CAD systems, one needs to undo the last of a trail before correcting the earlier ones.

(From Tom) On the bright side, although the crashes are annoying I have rarely lost work after a crash.
