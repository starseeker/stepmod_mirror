Naming of schemas

All ARM schemas shall have a name terminated with "_arm".

All MIM schemas shall have a name terminated with "_mim".

The schema name shall start with an uppercase letter.

 

Naming of entity data types and named types

All entity data types defined in the ARM of application modules shall be named with an uppercase first letter.

All entity data types defined in the ARM of application modules shall be named with a lowercase first letter.

All types, in ARMs and MIMs, shall be named with a lowercase first letter.

 

It is allowed, although not recommended, to name an ARM entity with the name of a resource entity.

Note: Express language is case-insensitive. Where needed, disambiguation between names of ARM and MIM entities can be done, prefixing the entity name with the schema name.

 

Text definitions of Select types    (!!! Look for updates !!!)

- for a non extensible SELECT type:

"

The <b>select_type_name</b> type is a list of alternate data types. It provides a mechanism to refer to an instance of one of these data types.

"

 

- for an extensible non-empty SELECT type:

"The <b>xxx_select_type_name</b> type is an extensible list of alternate data types. It provides a mechanism to refer to instances of the data types included in the types that extend the <b>xxx_select_type_name</b> type or in its extensions.

 

NOTE The list of entity data types may be extended in application modules that use the constructs of the present module.

"

 

- for an extensible empty SELECT type:

"The <b>xxx_select_type_name</b> type is an extensible list of alternate data types. It provides a mechanism to refer to instances of the data types included in the types that extend the <b>xxx_select_type_name</b> type.

NOTE  This empty extensible select requires extension in a further module to ensure that all entities have at least one valid instantiation.

"

- for an extended not extensible SELECT type:

"

The <b>yyy_xxx</b> type is an extension of the <b>xxx</b> type. It adds to this latter type, the entity data type <b>yyy</b>. 

"

 

- for an extended and  extensible SELECT type:

"

The <b>yyy_xxx</b> type is an extension of the <b>xxx</b> type. It adds to this latter type, the entity data type <b>yyy</b>. 

 

NOTE The list of entity data types may be extended in application modules that use the constructs of the present module.

"

Select types - automatically generated descriptions

It is highly recommended, for new modules, to use the XML repository functionality of automatic generation of descriptions of Select types.

To get the XSL to document your select types, add this attribute describe.selects="YES  to the ext_descriptions element in the arm_descriptions.xml and mim_descriptions.xml files 
e.g.

<ext_descriptions
        module_directory="activity_as_directed"
        schema_file="arm.xml"
        rcs.date="$Date: 2002/06/17 08:35:32 $"
        rcs.revision="$Revision: 1.7 $"
        describe.selects="YES"
>

 

Entity definitions 

See SC4 editorial directives (SC4 N1217) for more complete directives.

Reuse from AP214, PDM Schema UG, etc.

In the balloted version of the modules, the definitions of the ARMs were very poor. 

In order to improve them, do not hesitate to copy text definitions, rules, notes and examples that exist in the IRs (2nd ed). A lot of notes and examples have been added in order to clarify concepts that were not clear in the 1st ed. of the IRs. 

Other sources of texts are PDM UG 4.3 and AP214.

 

Normative versus informative text

Normative text shall only contain statements of fact (a xxx is..., a yyy defines ...) or constraints (a xxx shall refer to ...).

Text that provide recommendations, information about intended use, about possibilities (e.g. any sentence like "a xxx may ..." ) shall be embedded in notes.

In there are several notes in the same clause (4.2.x or 5.2.X ), they shall be numbered (starting from 1 at the first note in the clause).

English abbreviations are forbidden

Use of abbreviations "e.g." or "i.e. is not allowed by the SC4 directives. "e.g." shall be replaced by the expression "for example".

 

Where to include extension of Select types

In order to maximize reusability of modules, the Select types shall be extended as late as possible in the module arborescence.

For example, if the entity Activity may have documents associated to it without that requirement be "crucial" for the definition of the concept of activity, then extension of the extensible select type document_element_select (that provides the capability to associate a document to an entity) shall be done preferably in the AP module and not in module Activity or any other low level module.

On the other hand, in order to mention that such a requirement might be encountered, a note should be included in module Activity, explaining that it can be covered with an extension of the extensible select type document_element_select.

Note: inclusion of such an informative to a construct defined in another module requires to create a bibliography entry to this module, in the module that contains the note.

 

Mapping-table - How to document the mapping of an attribute that is an extensible Select type 

NOTE we assume in the example that the MIM element classified_select is an empty extensible select data type.

By analogy , in the application element column, treat the a to b form as a path in the ARM, and terminate on the ARM select.  Select types appear naturally in paths, but normally they must be followed by some entity.

Thus our table is now  : 

 
Application element MIM element Source Rules Reference path
CLASSIFICATION applied_classification_assignment 1054   applied_classification_assignment=>
classification_assignment
Classification to Classified_select
(as classified)		 PATH    

classification_assignment.item[i]->
classified_select
Classification to class
(as classifier)		 PATH     classification_assignment.classifier->class

 

 NOTE  It is conventional (see 10303-41Ed2  F.3) to name the subtype to be applied_classification_assignment instead of classification, and to name the select classification_item instead of classified_select. 

 

Mapping-table - How to document the mapping of an attribute that is an extended Select type

The reference path shall show the extension, using the symbols +> or <+ .

Example  :

specific_MIM_entity => general_MIM_entity

general_MIM_entity.item -> general_select

general_select +> specific_select 

 

Content of mapping-tables

In a mapping-table, the Application element column shall contain:

  • the name of an ARM entity(uppercase)
  • the name of an attribute if this attribute is of a predefined type (STRING, INTEGER, ....). In that case, the AIM element column contains the name of the corresponding attribute in the MIM. The source column shall contain the number of the part where the MIM entity is defined. If the MIM attribute is not an attribute of the MIM entity that maps the ARM entity, then the reference path column shall depict how to reach the MIM attribute from the MIM entity that maps the ARM attribute
  • an expression "<arm_entity_name1> to <arm_entity_name2> (as <attribute_name>) if the attribute attribute_name refers to an entity. In that case, the AIM element shall contain PATH and the source column shall be left empty. In such a case, the reference path column shall depict the entity sequence from the MIM entity that maps arm_entity_name1 to the MIM entity that maps arm_entity_name2.

    nb: in the case, attribute_name refers to a SELECT type, a distinct mapping shall be provided for every entity types in the SELECT type.
    => example of wrong Application element column: module Independent_property_representation
    => example of wrong Application element column and reference path column: module Product_categorization (Product_category_hierarchy)

When there are several alternate possibilities in the AIM element column, they shall be numbered and documented in the column Application element.

=> For example, in module Date_time, it should be explained, in the mapping of the entity date_time when date_and_time shall be used and when calendar_date should be used.

However, in that particular case, the mapping onto calendar_date is simply wrong as it does not enable to map all the attributes of date_time (so, only date_and_time shall be used).

 

Comments in Express files

The arm.exp and mim.exp files should include the CVS $Id: additional_module_guidelines.htm,v 1.7 2002/06/17 08:35:32 goset1 Exp $ keyword and the SC4 N number as a comment on the first few lines of the file.

The Id will get expanded and updated every time you check in the file to CVS. That way you can immediately tell when the file was last updated and will get around the issue raised by Greg about multiple versions of express files with the same N number.

E.g. 

(*
  $Id: additional_module_guidelines.htm,v 1.7 2002/06/17 08:35:32 goset1 Exp $
  N773 - ISO/CD TS - 10303-1042 Work request - EXPRESS MIM
*)

 

SCHEMA work_request_mim;

 

We should also delete the last update comment at the bottom of the file as this is redundant - the update information is held by the $Id: additional_module_guidelines.htm,v 1.7 2002/06/17 08:35:32 goset1 Exp $ field.

 

Inserting Notes and examples in xml files

In order to get the following output:

NOTE 1    This definition may be the basis for actions or the solution for action requests.

 

Insert:

 <note number="1">This definition may be the basis for actions or the solution for action requests. <note>

 

NB: The id attribute for note and example are identifiers that provide an anchor for the note/example. This enables hrefs to be set up by module_ref.

 

For example to set up a link to note 1 in the module activity_method, in the section on entities, use:

  <module_ref
       linkend="activity_method:4_entities:note:1">
      see note 1
  </module_ref>
 

 

This will result in:

  see note 1

NB:  note that "see note " is a HREF.

 

Example or note for a term definition

In order to see an example or a note that completes the definition of a term in clause 3, the <example> or <note> element shall be embedded in the <def> element.

 

Rule propositions

If a unique or where rule contains an expression, then it is a Formal Proposition.

If they do not contain an expression but have a description, then they are Informal Propositions.

E.g.

<where
        label="WR1"
        expression="SELF.identifier.role = 'id owner'">
        <description>The <b>identifier</b> role shall be 'id owner'.</description>

</where>

is a formal proposition, whereas

      <where
        label="document" >
        <description>The item plays the role of a document.</description>
      </where>
     

is an informal proposition.

 

Short names

If any MIM entities are defined in a module then shortnames need to be generated using the shortname generation tool at http://www.steptools.com/sc4/services/index.html

 

DTD for shortnames:

<!ELEMENT  mim          (express-g, shortnames?)                     >  
<!ELEMENT  shortnames   (shortname+)                                 >
<!ELEMENT  shortname    EMPTY                                        >
<!ATTLIST  shortname    name NMTOKEN #REQUIRED
                        entity NMTOKEN #REQUIRED                     >

 

An example of the XML is given in activity/module.xml

 <mim>
   <express-g>
     <imgfile file="mimexpg1.xml"/>
     <imgfile file="mimexpg2.xml"/>
   </express-g>
   <shortnames>
     <shortname name="APACAS" entity="applied_action_assignment"/>
   </shortnames>
 </mim>

 

This will result in a table in Annex A.

 

Express-G diagrams

The Express-G diagrams shall contain:  

- for each ARM and MIM, one or more schema level diagrams  

- for each schema that contains type or entity declaration, one or more entity level diagrams,  

- for each schema level diagram:    

  • a rectangle of a used-from module schema shall hide a hyperlink to the schema level diagram of that schema     
  • a rectangle of a used-from resource schema shall hide a hyperlink to the .xml file of that schema (as we do not have any ExpG for resource schemas)             
  • the rectangle of the module being considered shall hide a hyperlink to the first entity level schema diagram when it exists or to the MIM short listing in case where the MIM only contains USE-from statements 

- for each entity level diagram:     

  • a rectangle of a construct defined in the considered module hides a hyperlink to the specification of that construct (in arm.xml or mim.xml)     
  • a rectangle of a used-from construct hides a hyperlink to the specification of that construct (in arm.xml or mim.xml) in the module where that construct is defined (and not a hyperlink to the use-from statement)

 Titles for Express-G diagrams

Titles for diagrams are constructed from attributes  in the imgfile.content tag in armexpg1.xml, etc.

The xsl creates titles automatically  if  imgfile.content tag has a file attribute. The file tag also enables navigation of diagram pages

<imgfile.content
     module="activity"
     file="armexpg1.xml"

 The old method used an explicit title tag.
<imgfile.content
     module="activity"
     title="Figure C.1 - ARM Schema level EXPRESS-G diagram (1 of 1)">

Templates for Express-G title (not needed if imgfile.content tag has file attribute).

The title attribute of the element imgfile of the file armexpg1.xml should be:

Figure C.1 - ARM schema level diagram (1 of 1)

The title attribute of the element imgfile of the file armexpg<n>.xml should be:

Figure C.n - ARM entity level diagram (p of q) 

where p=n -1 and q= number of entity diagrams

The title attribute of the element imgfile of the file mimexpg1.xml should be:

Figure D.1 - MIM schema level diagram (1 of 1) 

The title attribute of the element imgfile of the file mimexpg<n>.xml should be:

Figure D.n - MIM entity level diagram (p of q) 

where p=n -1 and q= number of entity diagrams

NOTE  I removed the word Express-G in the titles because annex C and annex D introductions specify that the diagrams are Express-G and because SC4N1217 does not require this word for ARM diagrams of APs.

 

USE FROM statements in MIMs

For modularization methodological reasons, a MIM shall contain a USE FROM statement of the MIM schema of every module whose ARM is imported, through a USE FROM, in the ARM. This USE FROM statement shall not contain any list of imported construct.

In addition, a MIM may contain USE FROM statements of common resources schema. These USE FROM statements may contain a list of imported constructs.

Module renaming

To rename a module. 

1) create a new module using mkmodule script.

2) move the module.xml, express, expressg, arm.xml, mim.xml file from your old modules directory to new module directory.

DO NOT copy the sys directory or CVS directories

3) rename schemas in express and update files arm.xml, mim.xml

4) make sure that module.xml only refers to the new module name

5) Load the new module into Sourceforge: Check in your new module directory, sys directory and all module files

6) Delete ALL the files in your local old module and module/sys directory.  NOT the files in CVS directory! 

7)Under WinCVS, view the newly missing files, using View/File filter/Show missing

(This will display the list of files that is managed on Sourceforge site)

8) Select these files and click on Modify/remove selection

9) Commit to validate 

10) Tell everyone what the new module name is, so that they can change all references from the old module to new module

 

<module>  attributes: 

WGN numbers, part, version, status, checklists language, year of publicatio

Example of  attributes on module tag in module.xml

 
<module
   name="work_request"
   part="1042"
   version="1"
   wg.number="1149"
   wg.number.supersedes="722"
   checklist.internal_review="1062"
   checklist.project_leader="1063"
   checklist.convener="1064"
   status="TS"
   language="E"
   publication.year="2002"
   published="y"
   rcs.date="$Date: 2002/06/17 08:35:32 $"
   rcs.revision="$Revision: 1.7 $">
 

 

Where:
  • name is module name
  • part is the ISO 10303 part number.
  • version is the version of the model,
    1 the first edition,
    2 for a second edition or Tech Corr against 1, etc.
  • wg.number is the WG12 N-number of the document.
  • wg.number.supersedes is previous N-number  (optional, use  if previously been balloted - i.e. had a WG12 number assigned)
  • checklist.internal_review  is the WG12 N number of the internal review checklist 
  • checklist.project_leader   is the.WG12 N number of the project_leader  checklist
  • checklist.convener is the WG12 N number of the convener  checklist
  • status the ISO status (should be TS for PDM modules)
  • language     the language of the standard. Defaults to E - English
  • publication.year   (should be 2002 for PDM modules).
  • published     y if the module has been published as an ISO standard ( should be y 2002 for PDM modules) . 
  • rcs.date     Date attribute for source control software like RCS or CVS. E.g. $Date: 2002/06/17 08:35:32 $
  • rcs.revision     Revision attribute for source control software like RCS or CVS. E.g. $Revision: 1.7 $
For any module that has previously been balloted (i.e. had a WG number assigned) you should add the attribute
wg.number.supersedes with the old WG number to the module element in the module.xml file.

CVS tips

When you do a CVS update you get a stream of messages telling you about the progress.
It is worthwhile looking at these as sometimes CVS is a bit too clever.
 
One situation is when:
 
you check out a file, then modify it on your local machine.
 
In the meantime, someone else, checks out the same file, modifies it and checks it back in.
 
The SourceForge server now contains a file that is at a later revision than the one on your machine.
 
Next time you do a CVS update, CVS will recognize this and try to be smart.
It will try to merge your file with the file on the sever. i.e. CVS will modify your local copy.
If this happens, CVS will either merge the revisions to it satisfaction or will identify a conflict. In these cases you will see messages like this:
 

cvs server: Updating data/modules/contract

RCS file: /cvsroot/stepmod/stepmod/data/modules/contract/arm.xml,v
retrieving revision 1.8
retrieving revision 1.11
Merging differences between 1.8 and 1.11 into arm.xml
rcsmerge: warning: conflicts during merge
cvs server: conflicts found in data/modules/contract/arm.xml
C data/modules/contract/arm.xml
