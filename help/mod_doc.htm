Module

The top level element is <module> as defined by: 

<!ELEMENT module (keywords, abstract?, comments_to_reader?, contacts, purpose, 
 inscope, outscope?, normrefs?, definition*, abbreviations?, arm, arm_lf?, 
 mapping_table?, mim, mim_lf?, refdata?, usage_guide?, bibliography?)>
<!ATTLIST module
	xmlns:xlink CDATA #FIXED "http://www.w3.org/1999/xlink"
	name NMTOKEN #REQUIRED
	name.french  CDATA #IMPLIED
	part NMTOKEN #REQUIRED
	version NMTOKEN #REQUIRED
        sc4.working_group (3 | 12) "12"
	wg.number CDATA #REQUIRED
	wg.number.supersedes CDATA #IMPLIED
	wg.number.arm CDATA #REQUIRED
	wg.number.mim CDATA #REQUIRED
	wg.number.arm.supersedes CDATA #IMPLIED
	wg.number.mim.supersedes CDATA #IMPLIED
	wg.number.arm_lf CDATA #IMPLIED
	wg.number.mim_lf CDATA #IMPLIED
	wg.number.arm_lf.supersedes CDATA #IMPLIED
	wg.number.mim_lf.supersedes CDATA #IMPLIED
	wg.number.ballot_package CDATA #IMPLIED
	wg.number.ballot_comment CDATA #IMPLIED
	checklist.internal_review CDATA #IMPLIED
	checklist.project_leader CDATA #IMPLIED
	checklist.working_group CDATA #IMPLIED
	checklist.convener CDATA #IMPLIED
	status (CD | FDIS | DIS | IS | CD-TS | TS | WD) "CD-TS"
	publication.year CDATA #REQUIRED
	publication.date CDATA #IMPLIED
	published (y | n) "y"
	language (E) "E"
	previous.revision.year CDATA #IMPLIED
	previous.revision.cancelled (NO | YES) #IMPLIED
	revision.complete (NO | YES) #IMPLIED
	revision.scope CDATA #IMPLIED
	rcs.date CDATA #REQUIRED
	rcs.revision CDATA #REQUIRED
        development.folder CDATA #IMPLIED
>
Where:
  • part is the ISO 10303 part number.
  • version is the version of the model,
    1 the first edition,
    2 for a second edition or Tech Corr against 1, etc.
  • sc4.working_group is the SC4 working group responsible for the module.
  • wg.number is the SC4 N-number or document number for the module.
  • wg.number.supersedes is the SC4 N-number or document number that this module supersedes.
  • wg.number.arm is the SC4 N-number or document number for the arm.exp file of the module.
  • wg.number.mim is the SC4 N-number or document number for the mim.exp file of the module.
  • wg.number.ballot_packageis the SC4 N-number or document number for the ballot package in which this module was balloted.
  • wg.number.ballot_commentis the SC4 N-number or document number for the ballot package in which this module was balloted.
  • checklist.internal_reviewis the SC4 N-number or document number for the internal checklist of this module.
  • checklist.project_leaderis the SC4 N-number or document number for the project leader checklist of this module.
  • checklist.conveneris the SC4 N-number or document number for the convener checklist of this module.
  • status the ISO status
  • publication.year
  • published y if the module has been published as an ISO standard
  • language the language of the standard. Defaults to E - English
  • rcs.date Date attribute for source control software like RCS or CVS. E.g. $Date: 2005/10/07 15:51:19 $
  • rcs.revisionRevision attribute for source control software like RCS or CVS. E.g. $Revision: 1.12 $

Purpose

The element <purpose> is defined by: 
  <!ELEMENT purpose (%text-elts;)*>

It is used to document the purpose of the module and serves as the introduction to the module.

Inscope

The element <inscope> is defined by: 
  <!ELEMENT inscope (li+)>
  <!ELEMENT li (%text-elts;)*>
This element is used to define the scope of the module.
It is a list with each element in the list marked up with the <li> element.

Outscope (optional)

The element <outscope> is defined by: 
  <!ELEMENT outscope (li+)>
  <!ELEMENT li (%text-elts;)*>
This element is used to define explicitly what is out of scope of the module.
It is a list with each element in the list marked up with the <li> element.

The outscope element is optional so if the module has no out of scope statement, then omit the outscope element.

ARM

The Application Reference Model (ARM) of a module (or Clause 4 Information requirement in the module documentation) is defined by the element <arm>: 
  <!ELEMENT arm (uof+, uoflink?, express-g)>
Where:
  • The element <uof> describes the ARM Units of functionality.
  • The element <uoflink> describes the Units of functionality from other modules that have been interfaced to.
  • The element <express-g> identifies the ARM EXPRESS-G diagrams.

ARM UOF

The Units of functionality of the module are defined by the element <uof>. 
  <!ELEMENT  uof        (description, uof.ae+)                       >
  <!ATTLIST  uof
             name       NMTOKEN                            #REQUIRED >
Where:
  • the attribute name is the name of the unit of functionality. The name provides an identification that is used to link to the unit of functionality from another module through the use of the <uoflink> element.
  • the <description> element provides the description of the unit of functionality.
  • the <uof.ae> elements identify the ARM EXPESS entities that make up the unit of functionality.

ARM Foreign UOF

Any units of functionality from other modules that have been interfaced to by USE FROM constructs in the EXPRESS are defined by: 
  <!ELEMENT  uoflink    EMPTY                                        >
  <!ATTLIST  uoflink
             module     CDATA                            #REQUIRED >
             uof        CDATA                            #REQUIRED >
Where:
  • the attribute module is the name of the module in which the unit of functionality being included has been declared.
  • the attribute uof is the name of the unit of functionality being included. Note: This is the name declared in the <uof> element. It is case sensitive.

EXPRESS-G

The element <express-g> is defined by: 
  <!ELEMENT express-g (imgfile+ | img+)>
This element identifies a list of EXPRESS-G diagrams. Each diagram is defined by the element <imgfile>. This results in the EXPRESS-G diagrams being displayed in a new browser window.

Mapping_table

The mapping for the ARM to MIM is defined by the <mapping_table> element which comprises of set of application element mappings, each of which is defined by the element <ae>. This represents a row in the mapping table. 
  <!ELEMENT  mapping_table (ae+)                                   >
  <!ELEMENT  ae           (%mapping;, aa*)                         >
  <!ATTLIST  ae
             entity     NMTOKEN                            #REQUIRED>

Where:

  • <entity> is the name of the application element entity being mapped;
  • %mapping; is a specification of the mapping. aa is the definition of the mapping of the application element attributes.

The attribute mapping specification is defined by: 

  <!ELEMENT  aa           (%mapping;)                                  >
  <!ATTLIST  aa
             attribute  NMTOKEN                            #REQUIRED 
             assertion_to NMTOKEN                          #IMPLIED  >
Where:
  • attribute is the name of the attribute being mapped.
  • assertion_to
    • the mapping assertion. This will be placed in parentheses in the table.

Mapping specification

The mapping specification row is defined by: 

  <!ENTITY % mapping     "alt?, aimelt, source, express_ref*, refpath?" >

Where:
  • alt indeicates an alternative mapping.
  • aimelt is the AIM element that is being mapped to.
  • source is the source of the AIM element.
  • express_ref is a reference to a rule. See express_ref for details.
  • refpath is the reference path.

Reference data (optional)

The element <refdata> is defined by: 
		
<!ELEMENT refdata (%description.content; | refdata_subclause)*>
<!ELEMENT refdata_subclause (%description.content; | refdata_subclause)*>
<!ATTLIST refdata_subclause
	title CDATA #REQUIRED
>

The title of Clause 6 is automatically generated. The text explaining the diagram notation is also automatically generated. However, subclause titles are specified by the module developer. There should be one for each entity type for which Reference data is being defined, whether in the ARM or MIM. The title for the subclause should be "Classifications of <entity name>" with the case of the entity name preserved. The first part of the subclause should contain diagrams showing all of the taxonomy related to the entity type.

Within the subclause for each entity type, there should be a subclause for each URI defined as part of the taxonomy of that entity type. So, for an entity type named "Product", mapped to "product", which was the only entity type for which reference data "Tool" and "Part" was being defined the following clauses would appear:

  • 6.1 Classifications of Product - ARM-related diagrams go here
  • 6.1.1 Tool - URI and text are defined within this subclause
  • 6.1.1 Part - URI and text are defined within this subclause
  • 6.2 Classifications of product - MIM-related diagrams go here and either 6.2.1 and 6.2.2 similar to 6.1.1 and 6.1.2 or a statement that the same classifications for "Product" are applied to "product".

Usage_guide (optional)

The element <usage_guide> is defined by: 
<!--                    usage guide                                -->
<!ELEMENT usage_guide (%description.content; | guide_subclause)*>
<!ELEMENT guide_subclause (%description.content; | guide_subclause)*>
<!ATTLIST guide_subclause
	title CDATA #REQUIRED
>

The title of the usage guide annex is automatically generated. However, subclause titles are specified by the module developer.

Bibliography (optional)

The element <bibliography> is defined by: 
<!--                    bibliography                               -->
<!ELEMENT bibliography (bibitem.inc | bibitem)+>
<                    bibiographic item                          -->
<!ELEMENT bibitem (%bib-elts;)*>
<!-- an identifer for the bibliographic item.  It should be unique to the repository -->
<!-- If n then the bibliographic item has not been published                     -->
<!ATTLIST bibitem
	id ID #IMPLIED
        published (y | n) "y" >

<!-- A list of bibitems that can be included in module.xml -->
<ELEMENT bibitem.list (bibitem+)>

<!ELEMENT bibitem.inc EMPTY>
